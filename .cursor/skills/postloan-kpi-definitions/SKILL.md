# 贷后 KPI 口径（边界与校验）

## 触发条件（必须调用）

当请求满足任一条时必须调用本 skill：

- 解释或核对 `overdue_rate` / `dpd30_rate` / `recovery_rate` 的定义
- 解释 DPD 时序（`DPD2/3/5/15/30+`）与 vintage 曲线含义
- 排查报表口径不一致（分子/分母、件数/金额、模块混用）
- 解释或校验状态桶（`c/m0/m2/m3/m4`）及迁徙含义
- 解释巡检过程指标（`cover_rate`、`case_connect_rate`、`caseload` 等）
- 做 DPD30 月环比拆解（shift-share：结构效应 / 风险效应）

## 排斥条件（绝对不要调用 / 不要输出确定结论）

出现以下情况时，不要“凭经验”给出确定答案：

- 未知数据来源（表名/脚本/口径文档缺失），且用户要求“给一个确定口径”
- 业务在讨论“策略效果归因/ROI/因果结论”，但缺少实验设计或对照信息（本 skill 只负责口径，不负责因果）
- 指标名称相同但模块未知（例如只问“dpd30”但不说明是触达出口、vintage 还是催收桶）

此时必须输出：缺失信息清单 + 你能给出的“候选口径集合”+ 如何验证。

## 成功标准（做到这里才算完成）

满足以下条件才算“已给出确定性结果”：

1. 明确了所属体系（触达策略 / vintage 风险 / 催收回收与过程）之一，且声明“不混用”
2. 明确了单位口径（件数 or 金额/本金）
3. 给出了分子、分母、时间点/观察窗口的定义
4. 给出至少 1 个可执行的校验方法（读哪张表/哪段 SQL/哪份权威文档）

## 与现实的接缝（防幻觉校验）

- **可信来源优先级**：
  1) 权威口径文档：`D:/09wiki/wiki/postloan_kpi_definitions/01_口径定义.md`  
  2) 项目内实际产出 SQL / 报表脚本（以代码为准）  
  3) 人的口头描述（最低优先级）
- **强制“我不知道”出口**：若无法定位到 1) 或 2)，必须明确说“我不知道当前项目最终采用的口径”，并给出需要用户提供的最小证据（表名/SQL片段/字段）。

## 权威口径位置（必须引用这里）

## 输出格式（固定）

1. **体系与范围**：属于哪一套体系、模块范围、件数/金额
2. **口径**：分子/分母/窗口
3. **解释**：业务含义（只解释，不做因果承诺）
4. **校验**：如何用权威文档或代码/数据验证
