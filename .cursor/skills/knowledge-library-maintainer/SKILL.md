# 知识库维护器

## 目标

以最低维护成本，让知识在多个项目中可复用、可追溯、可持续更新。

## 推荐目录（固定在 D 盘）

统一根目录：

- `D:/09wiki/`

推荐结构：

- `D:/09wiki/wiki/`（权威业务文档与口径文档）
- `D:/09wiki/index/`（跨主题索引）
- `D:/09wiki/changelog/`（按日期记录变更）
- `D:/09wiki/skills/`（可复用技能草稿或来源片段）

## 同步策略

1. 项目内 wiki 保留执行上下文（贴近代码与脚本）。
2. `D:/09wiki/wiki/` 作为跨项目权威知识源。
3. 出现重要更新时，同时执行：
   - 同步文档到 `D:/09wiki/wiki/`
   - 新增一条 changelog
   - 更新路由 skill 与索引映射

## 固定模板与语言要求（默认强制）

1. 文档与索引尽量以中文维护（标题、段落与字段解释优先中文）。
2. 目录与文件命名使用 `snake_case` 或数字前缀（避免空格与不稳定路径）。
3. 后续维护默认沿用以下模板文件与风格（中文、短句、可检索）：
   - 根目录说明：`D:/09wiki/README.md`
   - 总索引：`D:/09wiki/index/00_总索引.md`
   - 主题入口：`D:/09wiki/wiki/<domain>/00_入口.md`
   - 变更记录：`D:/09wiki/changelog/YYYY-MM-DD_<topic>.md`
4. 变更记录最小字段集合（不可省略）：
   - 变更内容
   - 原因
   - 影响范围

## 最小工具集收敛（默认自动执行）

目标：避免多个 skill/文档重复描述同一口径，导致漂移与冲突。

### 触发条件（满足任一条就必须收敛）

- 新增了一个 skill 或新增了一篇权威文档
- 同名指标在不同地方出现两套解释/公式
- 用户反馈“同一个问题不同回答”
- 路由命中文档后发现“观点一致但表述重复”

### 收敛规则（边界优于路径）

1. **单一事实源（SSOT）**：
   - 口径/定义类事实：以 `D:/09wiki/wiki/` 的权威文档为最终事实源
   - 代码实现：以项目仓库内脚本/SQL 为最终事实源
2. **skill 的定位**：
   - skill 只保留：触发条件 / 排斥条件 / 成功标准 / 校验方式 / 路由策略
   - skill 不复制长篇口径原文（避免双写）
3. **去重策略**：
   - 如果两个 skill 覆盖同一问题域：保留“边界更清晰、工具更少”的那个
   - 另一个改为“跳转/指向型 skill”（只负责路由到权威文档或主 skill），或标记为废弃
4. **冲突处理**：
   - 若权威文档与代码冲突：在知识库中明确写出冲突点与当前采用口径，并要求补充决策证据；在证据不足时必须走“停止并提示人工”出口

### 成功标准（收敛完成的定义）

一次收敛被视为完成，必须满足：

- 同一指标/同一主题在知识库内只有一个权威定义入口（SSOT）
- 路由 skill 能稳定路由到该入口（不超过 3 篇文档）
- 被废弃的入口有明确指向（指向新入口）且在 changelog 留痕

## 触发维护的条件

满足任一条件就必须维护：

- KPI 口径变化
- 表结构/字段语义变化
- 报表计算逻辑变化
- 出现高频新问题
- 现有路由无法命中正确文档

## 排斥条件（必须停止并提示人工）

当满足任一条时，不允许“直接写入权威文档”：

- 新口径涉及跨部门决策/风控口径变更，但没有明确决策记录或负责人确认
- 无法定位变更来源（没有 PRD/会议纪要/SQL/报表输出作为证据）
- 变更会影响线上指标对账，但没有回滚/对账方案

此时必须输出：需要谁确认、需要什么证据、以及最小回滚策略。

## 成功标准（维护完成的定义）

一次“知识库更新”被视为完成，必须同时满足：

1. `D:/09wiki/wiki/` 下的权威文档已更新或新增
2. `D:/09wiki/index/` 的索引/入口能路由到该文档
3. 若涉及问答路由或口径定义，相关 `.cursor/skills/*` 已同步更新
4. `D:/09wiki/changelog/` 新增一条变更记录（含：变更内容/原因/影响范围）

## 必做更新动作

每次触发后按顺序执行：

1. 更新权威 wiki 文档（`D:/09wiki/wiki/`）。
2. 更新索引文件（`D:/09wiki/index/` 或项目索引）。
3. 若映射变化，更新路由 skill（`collection-inspection-wiki-router`）。
4. 若口径/公式变化，更新定义类 skill（如 `postloan-kpi-definitions`）。
5. 新增变更日志，至少包含：
   - 日期
   - 变更文件
   - 变更原因
   - 影响范围

## 输出检查清单

当被问“知识库是否最新”时，必须输出：

1. 当前状态（已更新/未更新）
2. 未完成更新清单
3. 需要修改的具体文件路径
4. 验证步骤
