# Git 版本控制操作 Skill

> 当用户主动要求或达到重要里程碑时使用此 Skill

---

## 触发条件

| 触发场景 | 说明 |
|----------|------|
| **用户主动要求** | 用户说"commit"、"push"、"提交"、"上传到 GitHub" |
| **重要里程碑** | 完成重要功能、修复关键 bug、交付阶段性产物 |
| **每日工作结束** | 重要工作完成后提醒用户是否需要提交 |

---

## 执行流程

### 1. 检查当前状态

```bash
# 查看修改状态
git status

# 查看具体改动
git diff
```

### 2. 分析改动内容

根据改动内容，确定：
- 是否有新文件需要添加
- 是否有不必要的文件（应加入 .gitignore）
- 改动是否属于同一主题

### 3. 提交改动

```bash
# 添加所有修改的文件
git add .

# 或添加指定文件
git add <文件名>
```

### 4. 编写提交信息

遵循以下规范：

| 提交类型 | 前缀 | 说明 |
|----------|------|------|
| 新功能 | `feat:` | 新增功能 |
| 修复 | `fix:` | 修复 bug |
| 文档 | `docs:` | 文档更新 |
| 重构 | `refactor:` | 代码重构 |
| 优化 | `perf:` | 性能优化 |
| 测试 | `test:` | 添加测试 |

**示例**：
```
feat: 添加 Collection Inspection 巡检脚本

- 新增 run_inspection_all_in_one_v4_25.py
- 优化 DPD30 维度计算逻辑
- 添加日期参数支持
```

### 5. 执行提交

```bash
git commit -m "提交信息"
```

---

## 推送流程（需用户确认）

### 1. 推送前确认

在执行 push 前，必须询问用户：
- **提交信息是否正确？**
- **是否需要添加更多改动？**
- **确认推送后不可撤回**

### 2. 执行推送

```bash
git push
```

### 3. 验证结果

```bash
# 确认推送成功
git remote -v
git log --oneline -3
```

## 重要里程碑提醒

当完成以下场景时，主动提醒用户：

| 里程碑 | 提醒话术 |
|--------|----------|
| 完成一个功能模块 | "功能模块已完成，是否需要 commit 记录这个里程碑？" |
| 修复重要 bug | "Bug 已修复，是否需要 commit 记录此次修复？" |
| 交付阶段性产物 | "阶段产物已完成，是否需要 push 到 GitHub？" |
| 连续多次 commit 后 | "已有 N 次 commit，是否需要 push 到远程仓库？" |

---

## 注意事项

| 规则 | 说明 |
|------|------|
| **.gitignore 检查** | 确保不提交敏感文件（密码、token、数据文件） |
| **提交粒度** | 每次提交应该是完整的逻辑单元 |
| **信息清晰** | 提交信息要清楚表达"做了什么" |
| **先 pull 再 push** | 推送前先拉取远程最新代码，避免冲突 |

## 常用命令速查

| 操作 | 命令 |
|------|------|
| 查看状态 | `git status` |
| 查看改动 | `git diff` |
| 添加文件 | `git add .` |
| 提交 | `git commit -m "信息"` |
| 推送 | `git push` |
| 拉取 | `git pull` |
| 查看历史 | `git log --oneline` |
| 创建分支 | `git checkout -b 分支名` |
| 切换分支 | `git checkout 分支名` |
